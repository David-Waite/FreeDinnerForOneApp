1. Current State

PROBLEM WITH CURRENT APPROACH

Encryption is performed using a single app-wide secret (EXPO_PUBLIC_ENCRYPTION_KEY) bundled into the JavaScript build. The key is combined with the user's UID to produce a per-user key:

encryptionKey = `${userId}-${APP_SECRET}`

This is obfuscation, not true encryption. Anyone who decompiles the app binary can extract the secret and decrypt any user's data if they have Firestore read access.

2. Target State — True E2E Encryption

GOAL

Each user's data is encrypted with a key derived from their own password. The key never leaves the device. Neither the server, the app developer, nor anyone with Firestore access can read user data without the user's password.

Proposed Key Derivation

// On login / account creation:
salt         = user.uid (or a random salt stored in user doc)
masterKey    = PBKDF2(password, salt, { iterations: 100000, keySize: 256 })
encryptionKey = masterKey

// Store in secure storage (expo-secure-store), NOT AsyncStorage
await SecureStore.setItemAsync('encryption_key', masterKey)

How It Works

  1. Account creation / password change — derive the master key from the password using PBKDF2, store it in expo-secure-store on device.
  2. Login — after Firebase Auth succeeds, re-derive the key from the password and store it in secure storage. The password is then discarded from memory.
  3. Encrypt on write — when saving data to Firestore, encrypt with the master key from secure storage (same AES approach but with the user-derived key).
  4. Decrypt on read — when hydrating from Firestore, decrypt with the stored master key.
  5. Logout — delete the master key from secure storage.
KEY CONSIDERATION — PASSWORD RESETS

If a user resets their password, the old master key is lost. This means all previously encrypted data becomes unreadable unless we plan for this. Options:

  • Key escrow: Encrypt the master key with a recovery code shown once at signup. User stores it offline.
  • Re-encryption on password change: When the user changes their password (not resets), decrypt all data with the old key and re-encrypt with the new key in a batch operation.
  • Accept the trade-off: Warn users that a password reset will wipe their encrypted history. Simple but harsh.

3. Migration Strategy

Migration must be handled carefully because production users already have data encrypted with the old app-wide key. The migration will be done in three phases.

1
INTERIM — ADD NEW ENV KEY + FALLBACK DECRYPTION
CURRENT STEP

This is where we are now. A proper EXPO_PUBLIC_ENCRYPTION_KEY has been added to EAS environment variables. SyncService.ts has a FALLBACK_SECRETS array that can decrypt data encrypted with any of the old default keys.

  • New data is encrypted with the new env key
  • Old data is still decryptable via fallbacks
  • No user action required
↓
2
FULL MIGRATION — RE-ENCRYPT ALL DATA WITH USER-DERIVED KEYS
TARGET

When E2E encryption is implemented, a one-time migration runs on each user's device:

  1. On login, detect if user has documents with isEncrypted: true that use the old key format
  2. Decrypt each document using the old key (try ENV_SECRET then FALLBACK_SECRETS)
  3. Re-encrypt with the user's new password-derived master key
  4. Write the re-encrypted document back to Firestore
  5. Mark migration complete on the user's profile: encryptionVersion: 2

Migration Flow

async function migrateUserEncryption(userId, newMasterKey) {
  const userDoc = await getDoc(doc(db, "users", userId));
  if (userDoc.data()?.encryptionVersion === 2) return; // Already migrated

  // Migrate sessions
  const sessions = await getDocs(collection(db, "users", userId, "sessions"));
  for (const session of sessions.docs) {
    const data = session.data();
    if (!data.isEncrypted || !data.data) continue;

    // Decrypt with old key
    const decrypted = tryDecryptWithFallbacks(data.data, userId);
    if (!decrypted) { flag for manual review; continue; }

    // Re-encrypt with new user-derived key
    const reEncrypted = AES.encrypt(JSON.stringify(decrypted), newMasterKey);
    await setDoc(session.ref, { data: reEncrypted, encryptionVersion: 2 }, { merge: true });
  }

  // Repeat for weight_logs, routines...

  // Mark complete
  await setDoc(doc(db, "users", userId), { encryptionVersion: 2 }, { merge: true });
}

CLEANUP — REMOVE LEGACY ENCRYPTION CODE

Once all active users have migrated (track via encryptionVersion field on user docs), remove the dead code:

  • Remove EXPO_PUBLIC_ENCRYPTION_KEY env variable from EAS
  • Remove APP_SECRET constant from WorkoutRepository.ts
  • Remove ENV_SECRET and FALLBACK_SECRETS from SyncService.ts
  • Remove fallback decryption logic (the keysToTry loops)
  • Simplify encrypt/decrypt to only use the user-derived key from expo-secure-store
  • Remove the migration function itself

6. Open Questions

DECISIONS NEEDED BEFORE IMPLEMENTATION
  • How do we handle password resets? (recovery code vs. data loss vs. key escrow)
  • Should the migration run eagerly (all docs on login) or lazily (re-encrypt each doc as it's accessed)?
  • Do we need a server-side Cloud Function to track / enforce migration progress?
  • How long do we keep fallback decryption before removing it? (e.g., 3 months after 95% migration)
  • Should we encrypt workout templates (routines) with E2E as well, or keep them accessible for potential future sharing features?
  • Do we continue using CryptoJS or switch to a faster native crypto library (e.g., react-native-quick-crypto)?
